# 仕様書：手動テスト支援用エラー監視ラッパーツール（TestSniffer）

> **バージョン:** 1.2
> **更新履歴:**
> - v1.0 → v1.1: `node-notifier` 代替方針追加、ログ即時追記に統一、`dotenv` 依存明示、OS通知フォールバック追加、ログフォーマット具体化
> - v1.1 → v1.2: ブラウザ閉じ時の自動終了、ウィンドウリサイズ追従、ナビゲーションタイムアウト延長、`WATCH_PATHS` フィルタ追加、録画機能追加

---

## 1. 開発背景と目的

Web アプリケーションの手動テストにおいて、画面上に明示的なエラーが表示されない「API 通信の失敗（4xx/5xx 系エラー）」や「フロントエンドの JavaScript エラー（Console Error）」は、テスト実施者が目視で見落とすリスクが高い。

本ツールは、普段通り手動テストを行うだけで、裏側で発生したエラーをリアルタイムに検知・通知し、テスト品質の向上とバグ報告の正確性を担保することを目的とする。

---

## 2. システム概要

Node.js 環境上で Playwright を起動し、テスト対象のブラウザを制御する。ツール経由で起動したブラウザ上のコンソールログおよびネットワークイベントを常時監視し、エラーを検知した際にローカル環境（デスクトップ）に通知を行う。

---

## 3. 機能要件

### 3.1. ブラウザ起動・管理機能

- **手動操作モードの提供**
  - ブラウザはヘッドフル（`headless: false`）で起動し、ユーザーが自由に画面を操作できること。

- **ウィンドウリサイズ追従**
  - ブラウザウィンドウをリサイズした際、ページの表示領域がウィンドウサイズに追従すること。
  - Playwright の `viewport: null` を使用して固定ビューポートを解除すること。

- **起動時 URL 指定**
  - `.env` ファイルの `START_URL` キーから、テスト開始時の初期 URL を指定してブラウザを開くこと。
  - コマンドライン引数（`--url`）でも上書き指定できること。
  - 両方とも未指定の場合は `https://example.com` をデフォルト URL として使用すること。
  - 初期 URL へのナビゲーションのタイムアウトは **60 秒** とすること。
  - ナビゲーションに失敗した場合は `[WARN]` を表示してツールを継続すること（クラッシュしない）。

- **セッション維持（任意）**
  - 毎回ログインする手間を省くため、`.env` の `USER_DATA_DIR` キーで指定した Chromium プロファイルディレクトリを使用して起動できるオプションを持つこと。
  - `USER_DATA_DIR` が未指定の場合は通常の一時プロファイルで起動すること。

- **ブラウザ閉じ時の自動終了**
  - ユーザーがブラウザウィンドウを閉じると、ツール（Node.js プロセス）も自動的に終了すること。

### 3.2. コンソールエラー監視機能

- ページ内で発生した `console.error` を検知すること。
- 検知したエラーメッセージのテキストを抽出すること。
- `console.warn` は監視対象外とする（将来的なオプション化のみ考慮）。

### 3.3. API（ネットワーク）エラー監視機能

- ブラウザが発行した HTTP リクエストのレスポンスステータスコードを監視すること。
- ステータスコードが **400 以上（4xx, 5xx 系）** の場合にエラーとして検知すること。
- エラー発生時に以下の情報を取得すること。
  - リクエスト URL
  - ステータスコード
  - HTTP メソッド（GET / POST 等）

### 3.4. 通知・ロギング機能

- **リアルタイム通知（OS デスクトップ通知）**
  - エラーを検知した瞬間、OS 標準のデスクトップ通知（ポップアップ）を表示し、テスト実施者に即座に知らせること。
  - 通知ライブラリは `node-notifier` を使用する。ただし、`node-notifier` が動作しない環境（インストール失敗・OS 非対応）では、通知なしで後述のコンソール出力とログ保存のみで動作する **フォールバックモード** に自動切替えすること。フォールバック時は起動メッセージで「OS 通知は無効です」と表示すること。

- **コンソール出力**
  - ツールを実行しているターミナル上に、エラーの詳細をわかりやすく出力すること。
  - 出力フォーマットは「タイムスタンプ・エラー種別・内容」を含み、`picocolors` で赤文字などに色付けすること。

- **ログファイルへの即時追記**
  - エラーを検知した**その都度**、プロジェクトルート配下の `logs/error_YYYYMMDD.log` に詳細ログを追記すること。
  - ツール終了時にまとめて書き出す方式は採用しない（プロセス異常終了時のデータ消失を防ぐため）。
  - `logs/` ディレクトリが存在しない場合は自動生成すること。
  - ログフォーマット（1 エラーにつき 1 ブロック）:

    ```
    [YYYY-MM-DD HH:mm:ss] [ERROR TYPE]
    URL       : <検知したページのURL または リクエストURL>
    Method    : <HTTPメソッド（APIエラーのみ）>
    Status    : <ステータスコード（APIエラーのみ）>
    Message   : <エラーメッセージ（Consoleエラーのみ）>
    ----------------------------------------
    ```

### 3.5. 監視パスフィルタ機能

- `.env` の `WATCH_PATHS` キーにカンマ区切りでパスを指定すると、そのパスを含む URL で発生したエラーのみ通知・ログ出力すること。
- 複数パスを指定した場合、いずれか 1 つでも一致すれば検知対象とすること。
- `WATCH_PATHS` が未設定の場合は従来通り全 URL のエラーを検知すること（後方互換）。
- 起動時に監視パスの設定状態をターミナルに表示すること。

### 3.6. 録画機能

- ターミナルのキー入力でブラウザ画面の録画を制御できること。
- **`r` キー**
  - 押下した時点から録画を開始すること（ツール起動時からではない）。
  - 同一ブラウザ内に録画専用タブを新規作成し、現在のページと同じ URL を開くこと。
  - ターミナルに `[REC]` メッセージを表示すること。
  - 録画中もエラー検知・通知・ログ出力は継続されること。
- **`s` キー**
  - 録画を停止し、`logs/videos/rec_YYYYMMDD_HHmmss.webm` としてファイル保存すること。
  - 保存完了後に録画用タブを閉じ、元のタブに戻ること。
  - 非録画状態での押下はエラーにならないこと。
- **録画中にブラウザを閉じた場合**
  - 自動的に録画を停止し、`logs/videos/` へファイルを保存してからツールを終了すること。
- `logs/videos/` ディレクトリが存在しない場合は自動生成すること。

## 4. 非機能要件・環境要件

- **動作環境:** Node.js v18 以上がインストールされた環境（Windows / macOS / Linux）
- **主要ライブラリ:**

  | ライブラリ | 用途 |
  |---|---|
  | `playwright` | ブラウザ制御・イベント監視・録画 |
  | `dotenv` | `.env` ファイルの読み込み |
  | `node-notifier` | クロスプラットフォームのデスクトップ通知 |
  | `picocolors` | ターミナル出力の着色 |

- **パフォーマンス:** ログ監視によるブラウザ操作の遅延（レイテンシー）が、人間の体感レベルで発生しないこと。

## 5. 設定ファイル仕様

### `.env`（プロジェクトルートに配置）

```env
# テスト開始時に開く URL
START_URL=https://example.com/dashboard

# Chromium ユーザーデータディレクトリ（省略可）
# 省略時は一時プロファイルで起動
# USER_DATA_DIR=/path/to/your/chrome-profile

# 監視対象パスフィルタ（省略可、カンマ区切りで複数指定可能）
# 省略時は全 URL のエラーを検知する
# WATCH_PATHS=/api/v1,/dashboard
```

### `.env.example`（リポジトリに含める雛形）

```env
START_URL=https://example.com
# USER_DATA_DIR=
# WATCH_PATHS=/api/v1,/dashboard
```

## 6. 画面・インターフェース仕様（CLI）

### ツール起動コマンド

```bash

# URL を引数で上書きする場合
npm run start:watch -- --url https://staging.example.com
```

## 8. ユースケース（利用フロー）

1. **準備:** `.env.example` をコピーして `.env` を作成し、`START_URL` にテスト対象環境の URL を記載する。必要に応じて `WATCH_PATHS` で監視対象パスを絞り込む。
2. **起動:** ターミナルから `npm run start:watch` を実行する。Chromium ブラウザが自動的に立ち上がる。
3. **テスト実行:** 仕様書（テストケース）に沿ってブラウザ上で手動テストを進める。
4. **エラー発生時の挙動:** 画面の裏側で API が落ちたり JS エラーが出ると、OS 通知が届き（有効な場合）、ターミナルにエラー詳細が表示され、ログファイルに即時追記される。
5. **録画（任意）:** バグ再現を記録したい場合はターミナルで `r` キーを押して録画を開始し、`s` キーで停止・保存する。
6. **確認・報告:** テスト実施者は通知・ターミナル・ログファイル・録画ファイルを確認し、エラー詳細（URL・エラー文・動画）を不具合票（Jira / Redmine 等）に起票する。
7. **終了:** ブラウザウィンドウを閉じるとツールも自動的に終了する。
